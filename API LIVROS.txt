FASE 1 — Planejamento e Modelagem (TEÓRICA)

Essa API tem o objetivo de gerir empréstimos de livros de um projeto.
Para isso, teremos algumas entidades importantes:

Usuário
Livro
Empréstimo

O usuário pode se registrar, editar e deletar seu perfil, além de registrar, editar e deletar seus próprios livros (não pode fazer nada disso com outros usuários, mas pode visualizar o perfil e os livros de outros usuários).
O usuário irá conter: id, nome, email, senha_hash, created_at, updated_at.

O livro poderá ser registrado, editado ou deletado pelo seu dono (usuário com id correspondente) e irá conter: id, quantidade_total, quantidade_disponivel, user_id, nome, editora, comentario (até 200 caracteres), created_at, updated_at.

O empréstimo será uma entidade que conterá: id, livro_id, dono_id, receptor_id, data_inicio (data em que o empréstimo foi criado), data_fim (data limite acordada para devolução), data_devolucao (preenchida apenas quando o livro for devolvido), created_at.
O empréstimo possui 3 status:

Ativo (empréstimo em andamento, sem devolução)

Devolvido (livro devolvido dentro ou fora do prazo)

Atrasado (data de devolução ultrapassada sem devolver o livro)

Usuários:

O usuário pode criar, editar e deletar apenas o próprio perfil.
O usuário pode visualizar o perfil e os livros de outros usuários.
Um usuário pode emprestar vários livros.
Um usuário pode pegar livros emprestados de outros usuários.
Um usuário não pode pegar o próprio livro emprestado.

Livros:

Um livro pertence a um único usuário (dono).
Apenas o dono pode criar, editar ou deletar um livro.
Um livro só pode ser emprestado se estiver disponível (quantidade_disponivel >= 1 e disponivel = true).
Um livro pode se tornar indisponível pelo seu usuário dono (caso de alguma emergência).
O dono pode tornar um livro indisponível manualmente.
A quantidade disponível deve ser decrementada ao criar um empréstimo.
A quantidade disponível deve ser incrementada ao devolver um empréstimo.

Empréstimos:

Um empréstimo só pode ser criado se o livro estiver disponível.
Um empréstimo ativo impede outro se não houver quantidade disponível.
Apenas o receptor pode devolver o livro.
O status do empréstimo pode ser Ativo, Devolvido ou Atrasado.
Ao devolver o livro: o status deve ser atualizado para Devolvido e a data_devolucao deve ser preenchida.
Um empréstimo é considerado Atrasado quando a data atual ultrapassa a data_fim prevista e o livro ainda não foi devolvido.

ENDPOINTS
Users:
POST /usuarios
GET /usuarios
GET /usuarios/:id
PUT /usuarios/:id
DELETE /usuarios/:id

Livros:
POST /livros
GET /livros
GET /livros/:id
PUT /livros/:id
DELETE /livros/:id
GET /usuarios/:id/books

Empréstimos:
POST /emprestimos
GET /emprestimos
GET /emprestimos/:id
POST /emprestimos/:id/return
GET /emprestimos?user=1
GET /emprestimos?status=ATIVO


FASE 2 — Preparação do Ambiente e Estrutura do Projeto

Nesta etapa foi realizada a preparação do ambiente de desenvolvimento e a definição da estrutura inicial do projeto. O objetivo desta fase é criar a base necessária para o desenvolvimento da API, garantindo organização, separação de responsabilidades e facilidade de manutenção.

O projeto foi desenvolvido utilizando Node.js e o framework Express, seguindo o padrão de arquitetura em camadas.

O projeto foi estruturado com separação de responsabilidades, onde cada parte do sistema possui uma função bem definida. Essa abordagem facilita a compreensão do código, a manutenção e a evolução da aplicação.

A organização do fluxo da aplicação segue o seguinte modelo:

Recebemos uma requisição HTTP
Rotas -> definem as URLs e os métodos HTTP
Controllers -> recebem a requisição e retornam a resposta
Services -> contêm as regras de negócio
Models -> realizam o acesso e manipulação dos dados

Inicialização do Projeto

O projeto foi inicializado utilizando o gerenciador de pacotes npm, que é responsável por gerenciar as dependências da aplicação. A inicialização gerou o arquivo package.json, onde ficam registradas as informações do projeto e as dependências utilizadas.
npm int -y

Foram instaladas as seguintes dependências principais:

express: framework responsável por facilitar a criação do servidor HTTP e o gerenciamento de rotas.
dotenv: utilizado para o gerenciamento de variáveis de ambiente.

npm install express dotenv

Também foi instalada uma dependência de desenvolvimento:
nodemon: utilizado para reiniciar automaticamente o servidor durante o desenvolvimento, sempre que houver alterações nos arquivos.

npm instal -D nodemon

Estrutura de Pastas

A estrutura inicial do projeto foi definida da seguinte forma:

src/
controllers/
routes/
services/
models/
middlewares/
app.js
server.js

Essa organização permite que cada camada da aplicação tenha uma responsabilidade clara, evitando arquivos extensos e facilitando futuras alterações.

Separação entre app.js e server.js
O arquivo server.js ficou responsável apenas por iniciar o servidor HTTP e definir a porta de execução da aplicação.
O arquivo app.js ficou responsável por configurar a aplicação Express, registrar os middlewares e definir as rotas da API.
Essa separação facilita a reutilização da aplicação em testes automatizados e mantém o código mais organizado.

Configuração de Variáveis de Ambiente

Foi criado um arquivo .env para armazenar variáveis de ambiente, como a porta utilizada pelo servidor. Esse arquivo não deve ser versionado, garantindo maior segurança e flexibilidade entre ambientes de desenvolvimento e produção.

- Configuração do Express

O Express foi configurado para receber requisições no formato JSON, utilizando o middleware responsável por interpretar o corpo das requisições.
Uniformização - essa configuração é essencial para o funcionamento correto da API, já que os dados trafegam no formato JSON.

- Primeira Rota de Teste

Foi criada uma rota inicial apenas para teste de funcionamento da API. Essa rota permite verificar se o servidor está ativo e se a estrutura básica do projeto está funcionando corretamente.
A resposta dessa rota retorna uma mensagem simples em formato JSON, confirmando que a API está em execução.

Execução do Projeto

Foi configurado um script de desenvolvimento utilizando o nodemon, permitindo iniciar o servidor de forma automática durante o desenvolvimento.
Com essa configuração, o servidor foi iniciado com sucesso e a API respondeu corretamente às requisições de teste, confirmando a conclusão da Fase 2.
